# Attune — Roadmap

Ideas and directions for where to take this. Not prioritized yet — to be discussed.

---

## 1. Automatic goal inference

**The problem with manual goals:**
The current system requires users to write their goals in `goals.yaml`. This is friction, and people won't keep it updated as their life changes.

**The idea:**
Infer goals automatically from behavioral signals in email and calendar — what you write, who you reply to fast, what your calendar protects time for.

**Data sources:**
- **Sent emails** — what topics you initiate reveals more than what you receive
- **Reply speed** — fast replies signal important senders/topics
- **Calendar patterns** — recurring blocks, named milestones, protected time
- **Named events** — "PhD defense", "grant deadline" → explicit milestones

**Proposed flow:**

```
fetch last 30d of sent emails + calendar
        ↓
extract signals:
  - sent email topics
  - fast-reply senders (< 1h avg)
  - recurring calendar events
  - named milestones (deadline keywords)
  - protected time blocks
        ↓
LLM infers 3-5 natural language goals from signals
        ↓
user reviews + edits → saved to goals.yaml
```

**UX sketch (`attune goals --infer`):**

```
$ attune goals --infer

Analyzing last 30 days of email and calendar...

Inferred goals:
  1. Finishing PhD thesis — defense likely mid-2026
     Signals: 8 calendar blocks "thesis writing", fast replies to
     supervisor, 3 sent emails with "chapter" in subject

  2. Applying for postdoc / research positions
     Signals: 2 sent emails with CV attached, calendar "interview prep"

  3. Active side project with a collaborator
     Signals: recurring evening blocks, fast replies to collaborator

  4. Maintaining fitness routine
     Signals: "Swim" and "Gym" appear weekly in calendar

Accept? [Y]es  [E]dit  [R]egenerate
```

**Research angle:**
Goals are a latent variable inferred from observed behavior — same problem as preference inference in recommendation systems, but with richer signals (email content, reply timing, calendar structure). Potentially publishable on its own.

**Questions to discuss:**
- How much history to use? (30 days? 90 days?)
- How often to re-infer? (weekly prompt? change detection?)
- How to handle goal transitions gracefully (thesis done → new job)?
- Privacy implications of sending email content to an LLM for goal inference

## 2. Trained reranker (replaces prompted LLM)

**The problem with prompting:**
Each triage decision is one LLM call — slow (~2s), expensive (~$0.001), and the model wasn't trained for this task specifically.

**The idea (from initial discussion):**
Train a small cross-encoder model on (goal, email, relevance) pairs. The model takes both texts jointly and outputs a relevance score. Same structure as a reranker in RAG, but the "query" is your goals and the "document" is an email.

```
score = cross_encoder(goals, email)   # → 0.0 to 1.0
label = threshold(score)              # → URGENT / SOON / LATER / IGNORE
```

**Why this is better:**

| | Prompted LLM | Trained reranker |
|---|---|---|
| Speed | ~2s per email | ~10ms |
| Cost | ~$0.001 | ~$0.00001 |
| Trainable | No | Yes |
| Personalizable | Prompt only | Fine-tuned on your data |

**Training data:**
Bootstrap with synthetic (goal, email, label) triplets generated by an LLM, then improve with real user corrections over time. A few hundred examples might be enough to start.

**Research angle:**
Learned value-of-information estimation for personal communication. Combined with automatic goal inference, this is a full paper: *learn what you care about from behavior, then learn to filter your inbox accordingly.*

**Questions to discuss:**
- Which base model? (DistilBERT, MiniLM, something smaller?)
- How to collect corrections without friction?
- How to handle cold start (no user data yet)?
- Training pipeline — how often to retrain?

## 3. Real-time monitoring ✅

Implemented as `attune watch`. Polls every N minutes, triages new emails on arrival, and fires desktop notifications for URGENT.

**Future improvements:**
- Gmail push notifications via Pub/Sub (webhooks) instead of polling
- Mobile push notifications (phone, not just desktop)

---

## 4. Multi-channel

Same VoI judge, different connectors. Priority order (to discuss):
- Slack (most requested for knowledge workers)
- WhatsApp
- Outlook / Teams (relevant for university / enterprise users)

The triage core is channel-agnostic — only the connector changes.

---

## 5. Web onboarding

Current setup requires a terminal, YAML file, Google Cloud Console, and an API key. That's too much friction for anyone outside the two of us.

Minimum viable onboarding:
- Connect Gmail in 2 clicks (OAuth)
- Write or infer goals in a text box
- Get first digest in under 5 minutes

---

## 6. Feedback loop

Every wrong label is a training signal. The product needs a way to capture corrections:
- "This wasn't URGENT" → LATER
- "I always want to hear from this sender" → add to goals

Corrections feed into reranker retraining (ties back to item 2).
